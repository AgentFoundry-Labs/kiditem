# 쿠팡 WING 상품등록 재기획

현재 구현을 쿠팡 공식 등록 14단계 기준으로 감사한 결과와, 그 갭을 메우기 위한
실행 계획이다. **리버스 결과(폼 구조·API·DOM 함정)는 이 문서에 다시 쓰지 않는다** —
[`coupang-wing-registration-spec.md`](./coupang-wing-registration-spec.md) 가 그 권위다.
이 문서는 "무엇이 비었고 어떤 순서로 채울 것인가"만 다룬다.

> **라인 앵커 주의**: 아래 `file:line` 은 작성 시점 워킹트리 기준이다.
> `wing-registration-fill.js` 는 미커밋 수정분이 있어 수십 행씩 밀린다.
> 라인이 어긋나면 인용된 **심볼명으로 grep** 하라 (`rg -n "optionFailed" …`).

## 1. 요약

> **2026-07 라이브 실측 반영판.** 정상 등록·판매중인 자사 상품 1건을 수정 화면에서
> 전 섹션 실측하고, 저장소가 들고 있는 V4.6 엑셀 양식을 전수 파싱해 갱신했다.
> 실측 결과 자체는 [`spec §1`](./coupang-wing-registration-spec.md) · [`spec §9`](./coupang-wing-registration-spec.md) 에 있다.
> 이 문서는 **그 실측이 계획을 어떻게 바꾸는가**만 다룬다.

**쿠팡 14단계 중 두 경로를 합쳐도 완결되는 단계는 3단계(상품명·카테고리·상세설명)이고,
그마저 경로마다 갈린다.** 다만 실측으로 "0%인 6단계" 의 성격이 갈라졌다 —
배송·반품은 **채울 데이터가 없는 게 아니라 확인만 하면 되는 것**으로 내려갔고,
상품 속성(구 검색필터)은 **36개 전량 미입력**으로 올라갔다.

| 경로 | 진입점 | 채우는 것 | 못 채우는 것 |
|---|---|---|---|
| 엑셀 일괄 | `collected-products/page.tsx:178` → `generateWingExcelForCandidates` | 검색어·미성년자·부가세·병행수입·바코드·고시값·할인기준가 | **상세설명(항상 공란)**·**출고리드타임(필수)**·**인증·신고(필수 가능)**·상품속성 |
| 단일 직접등록 | `page.tsx:205` → `registerSingleProductToWing` → 확장 | **상세설명(CDN+HTML)** | 검색어·고시·부가세·미성년자·바코드·**제조사**·**등록상품명**·**정상가**·상품속성 |

배송·반품은 어느 경로에서도 "못 채우는 것" 이 아니다 — 아래 §3 B1/B2 참조.

### 실측이 뒤집은 판단

| 이전 판단 | 실측 결과 |
|---|---|
| 브랜드 = `해피프랜즈` (`excel.ts:166`, 주석에 "실제 등록 상품 기준") | **정반대.** 브랜드칸은 비고 `브랜드없음(또는 자체제작)` ☑, **`해피프랜즈` 는 제조사**다. 확장의 `selfMade` 체크가 맞았고 엑셀이 틀렸다 |
| 반품/교환 = `WING_COL` 75~86 미매핑 | **75~86 은 인증∙신고 등 정보** 4칸×3블록이다. 반품 컬럼이 아니다 |
| `emptyRow()` 가 템플릿 기본값을 지운다 | 양식 `기본` 시트에 **데이터행이 0건**이다. 지울 기본값이 없다 |
| 배송·반품 엑셀 컬럼 정의가 없다(=우리 누락) | 117컬럼 전수 확인 결과 **양식에 아예 없다.** 우리 누락이 아니라 양식 설계다 |
| `excel.ts:6` "업로드 폼에서 1회 설정" 은 미검증 전제(U1) | **검증 완료.** 양식 구조가 그 전제를 강제한다 |
| 검색필터 = 별도 선택 단계 | **폐지.** 상품 속성으로 통합됐다. 코드/문서의 "검색필터 단계" 개념 자체가 stale |
| 상세설명 최종 형태가 미검증 | **형태는 검증됨** — 판매중 상품이 `<center><img></center>` 한 줄로 저장돼 있다 |

### 그대로 유효한 진단

두 경로가 **서로 모순되는 값**을 넣는 문제는 실측으로 오히려 확정됐다. 엑셀은
브랜드=`해피프랜즈`/제조사=`kiditem` 을 쓰고, 확장은 `input[name="selfMade"]` 를 체크한 뒤
제조사를 비운다(`fill.js:617-620`). **라이브 기준으로는 확장 쪽이 맞고 엑셀이 틀렸다.**

그리고 확장 경로는 **필수값이 하나도 안 들어가도 `ok:true` 를 반환한다.**
실패 판정 조건이 `detailUploadError` 단 하나이기 때문이다(`fill.js:724-728`).
카테고리 미선택·옵션 미입력·판매가 0 이 전부 겹쳐도 웹에는 성공 토스트가 뜬다.

## 3. 블로커

쿠팡이 필수로 요구하는데 우리가 채우지 못하는 것. **왜 블로커인지 + 값이 어디서 와야 하는지.**

### B1·B2. 배송·반품 — **블로커에서 강등한다**

실측 두 가지가 이 항목의 성격을 바꿨다.

1. **값이 이미 판매자 계정에 설정돼 있다.** 출고지·반품지가 같은 법인
   `(주)거영아이앤디` 이고 택배사 `CJ대한통운`, 기본배송비 `3,000`, `30,000` 이상 무료,
   반품배송비 `3,000` 이 전부 잡혀 있다(`spec §1`). 상품별로 달라지는 값이 아니다.
2. **일괄등록 엑셀 양식에 배송·반품 컬럼이 아예 없다.** 117컬럼 전수 확인 결과
   배송 관련 컬럼은 `출고리드타임`(65) 하나뿐이다(`spec §9`).

→ 따라서 **"새 모델 + 설정 UI + 마이그레이션" 은 이 블로커를 푸는 데 필요하지 않다.**
필요한 것은 확장이 배송/반품 섹션의 현재 값을 읽어 **기대값과 일치하는지 확인하고,
다르면 `ok:false` 로 세우는 것**뿐이다(저장이 아니라 검증).

- **남는 진짜 갭**: `출고리드타임`(col65)이 양식상 **필수**인데 `buildProductRows` 가
  대입하지 않아 엑셀 전 행이 공란이다. 이건 상수 한 개면 끝난다.
- **미루는 것**: 기대값을 어디에 둘지. 지금은 단일 조직·단일 출고지라 판매자 프리셋
  상수로 충분하다. 조직이 둘 이상 되면 그때 `ChannelAccount`/`LegalEntity` 스코프로
  올린다 — 루트 `AGENTS.md` 의 조직 경계 계약상 **전역 리터럴로 굳히지는 말 것**.
  `LegalEntity.address`(core.prisma:268)는 존재하지만 읽는 코드가 저장소에 0건이다.
- **폐기된 근거**: 이전 판단이 근거로 삼은 "`WING_COL` 75~86 반품 미매핑" 과
  "`emptyRow()` 가 템플릿 기본값을 지운다" 는 **둘 다 사실이 아니었다**(§1 표).

### B3. 카테고리별 필수 옵션 속성

- **왜 블로커**: 카테고리는 상품별로 추론하는데(`resolveWingCategories`) **옵션명만 물총
  프리셋의 `색상`/`수량` 리터럴로 고정**된다(`flow.ts:100-103`). 열쇠고리·문구류로 추론돼도
  옵션은 색상/수량이 들어간다. 확장의 분기도 `String(opt.type).includes('수량')` 문자열
  휴리스틱 하나뿐이라 `개수`·`용량`·`사이즈` 는 전부 텍스트 옵션칸으로 간다.
- **⚠️ 실측이 만든 함정**: 라이브 기준 상품(열쇠고리)의 옵션명이 **마침 `색상`/`수량`** 이라
  하드코딩이 우연히 맞아떨어졌다. "동작한다" 는 근거로 쓰지 마라 — 맞은 것이지 옳은 것이 아니다.
- **더 나쁜 부분**: 확장은 옵션 **유형명을 폼에 쓰지 않는다.** placeholder(`옵션값 입력` /
  `숫자만 입력`)만 보고 값을 밀어넣으므로, 카테고리가 `사이즈`/`용량` 컬럼을 렌더해도
  **실패하지 않고 색상값이 그 칸에 들어간 뒤 `option:색상=혼합` 으로 성공 로그를 남긴다.**
  값 주입 전에 그 행의 옵션명 라벨을 읽어 `opt.type` 과 대조하는 가드가 필요하다.
- **값의 출처**: 카테고리별 속성 정의 API. **엔드포인트 미발견**(spec §3, `featureGroup` 은 A/B 플래그였음).
  우회로는 §6 참조.
- **주의**: `WingCategoryPreset.requiredPurchaseOptionTypes`(`excel.ts:134,165`)는
  **선언만 있고 읽는 곳이 없는 dead config** 다. 프리셋을 카테고리별로 늘려도
  `flow.ts:100-103` 리터럴을 같이 고치지 않으면 옵션은 그대로 물총이다.

### B3-b. 상품 속성 36개 전량 미입력 (구 검색필터)

- **왜 블로커**: 라이브 카테고리에 `입력 가능 상품 속성 열기 (36)` 이 있고 우리는 0개를 채운다.
  쿠팡이 폼에서 명시한 손실이 **최대 12% 추가 노출**이다. 필수 속성이 있는 카테고리면 반려 위험.
- **개념 정정**: 별도 `검색필터` 단계는 폐지됐다. 폼은 **상품 속성**, 엑셀 양식은
  **`검색옵션유형N`/`검색옵션값N`**(21~60) 으로 같은 것을 다르게 부른다(`spec §3`).
  자사 워크북 파서도 엑셀 헤더명을 읽으므로 **코드 용어를 `상품속성` 으로 리네이밍하지 마라** —
  외부 계약이 깨진다. 문서·주석에서만 "= 폼의 상품 속성" 을 병기한다.
- **값의 출처**: 유일 프리셋에 `defaultSearchOptions` 키가 없어 파이프라인이 끝에서 끝까지 죽어 있다.
  카테고리↔속성 매핑은 여전히 미확보지만, **속성 값 후보는 이미 저장소에 있다** —
  양식 `hidden` 시트의 `색상계열` 17종 등(`spec §9`).

### B4. 카테고리별 고시 항목 스키마

- **왜 블로커**: 고시 카테고리가 `'어린이제품'` 으로 고정되고(`excel.ts:154`), 값은 프리셋 7개 중
  첫 칸(품명)만 상품명으로 치환된다(`flow.ts:110-111`). 엑셀은 14칸을 지원하는데 7개만 채운다.
  **확장 경로는 고시값을 아예 안 쓴다(0/14).**
- **실측이 확정한 불일치**: 라이브 열쇠고리 상품의 고시는 **`기타 재화`** 다.
  카테고리는 `resolveWingCategories` 로 상품별 해석하면서 고시만 프리셋 고정이라
  **카테고리가 동적일수록 고시가 더 자주 틀린다.** 엑셀 경로는 이 값을 그대로 업로드한다.
- **값의 출처**: `notice/form?categoryId=` 응답. 그런데 이 `categoryId`(6827)는 우리가 가진
  `displayCategoryCode`(77390)와 **다른 체계이고 매핑 규칙을 모른다**(spec §2 "ID 가 두 종류다", 미해결).
- **부분 해소 가능**: KC 인증번호는 **이미 우리 DB에 있다**
  (`product-basics.presenter.ts:117-119`). 고시 스키마를 모르더라도 `'상세정보 별도표기'`
  리터럴을 실값으로 바꾸는 것은 지금 가능하다.

### B4-b. 인증∙신고 정보 — 빈칸은 "대상 아님" 이 아니다

- **왜 블로커**: 양식 col75/79/83 설명이 "정보가 없으면 **[인증대상아님]을 선택**" 이라고
  명시한다. `WING_COL` 에 75~86 인덱스가 아예 없어 전 행이 공란으로 나가고,
  `assertBaseSheetLayout` 도 이 구간을 검사하지 않아 조기 실패조차 없다.
  현재 프리셋이 `어린이제품`(KC 필수 계열)이라 실제 반려 위험이 있다.
- **값의 출처**: **재리버스 불필요.** 허용값 열거가 양식 `hidden` 시트에 있다(`spec §9`).
  기본값 `인증대상아님` 을 넣고 카테고리별 오버라이드만 프리셋에 두면 된다.

### B5. 재고수량 하드코딩 999

- **왜 블로커**: 실보유 0개여도 쿠팡에 999개로 노출된다 → 품절 미반영 주문 유입, 쿠팡 페널티.
  단일 SKU 고정이라 옵션별 차등도 구조적으로 불가능하다.
- **값의 출처**: `SellpiaInventorySku.currentStock`. **데이터도 읽기 포트도 이미 있다** —
  `SELLPIA_INVENTORY_SKU_READ_PORT` 가 `currentStock`/`purchasePrice`/`salePrice` 를 싣고
  `findByIds`/`findByCodes`/`findByBarcodes`/`findByNormalizedNames`/`search` 5종 조회를 제공한다.
  **sourcing 도메인이 이 포트를 구독하지 않을 뿐이다** → §4 D5.
- **완화**: 확장은 폼을 채우기만 하고 제출은 사람이 하므로, 999가 무인으로 게시되는 경로는
  엑셀 일괄업로드(사용자가 직접 업로드)뿐이다.

### B6. 판매가 — 폴백이 죽은 코드다

- **왜 블로커**: 감사가 처음에 "소스 2개(수기 입력 + `SourcingCandidate.sellPrice`)"로 봤으나,
  **`sellPrice` 컬럼은 `prisma/models/sourcing.prisma` 에 존재하지 않는다.**
  `sourcing-api.ts:514,544,557` 의 `p.sellPrice` 는 영구 `undefined` 이고,
  `flow.ts:94` 의 `detail.price_krw` 분기도 따라서 도달 불가다.
  즉 **살아있는 소스는 수기 `registrationInput.salePrice` 하나뿐이고, 비면 폴백 없이 0원**이다.
  `normalizePrice` 는 하한이 0이라 예외도 안 던진다. 엑셀은 `'0'` 을 기입하고,
  확장은 `Number(salePrice) > 0` 단락으로 **판매가 칸을 조용히 건너뛴다.**
  카테고리는 확신 낮으면 등록을 막는데(`flow.ts:147,190`) 가격에는 같은 가드가 없다 — **정책 비대칭**.
- **값의 출처**: `SellpiaInventorySku.salePrice`/`purchasePrice`(B5와 같은 포트).
  원가·환율 재료도 있다(`rocket-pricing.ts` `CNY_TO_KRW_RATE=190`,
  `option-pricing-resolver.ts`) — 다만 전부 salePrice 를 **소비**하는 방향이고 생산하지 않는다.

### B7. 확장 경로의 무음 성공 (`ok:true` 오보)

- **왜 블로커**: 위 모든 갭을 *보이지 않게* 만드는 증폭기다. 실패 판정이 `detailUploadError`
  하나뿐이라(`fill.js:724-728`), 카테고리 미선택 + 옵션 0개 + 판매가 미입력이 동시에 나도
  사용자에겐 "자동 입력을 시작했어요" 성공 토스트가 뜬다(`page.tsx:206`).
  `steps` 배열에 단서가 남지만 웹 응답 타입(`flow.ts:199`)에서 소거되어 도달하지 않는다.
- **값의 출처**: **신규 데이터 불필요.** 확장이 이미 `steps` 를 반환하므로 판정 규칙과
  응답 타입만 넓히면 된다. 갭 대비 비용이 가장 낮은 항목이다.

### B8. 등록 결과 영속화 부재

- **왜 블로커**: WING 등록 시도가 우리 DB에 아무 흔적도 남기지 않는다
  (`flow.ts:175-207` 은 `{ detailImage }` 만 반환). 재시도·중복등록 방지·성공률 추적이 전부 불가능하다.
  제출 자동화(14단계)를 넣기 전에 반드시 선행돼야 한다 — 자동 제출 결과가 유실되면 복구 불가다.
- **값의 출처**: **인프라는 이미 있다.** `ProductPreparation` 상태머신
  (`product-registration.service.ts` 의 `createDraft`/`submit`/`cancel`/`fail`)이
  다른 버튼에서 이미 가동 중이고, 웹 클라이언트도 `sourcing-api.ts:655` 에서
  `POST /api/sourcing/candidates/{id}/preparations` 를 호출한다.
  **WING 경로만 이 배선을 우회**한다. 도메인 계약은 `collected-products/AGENTS.md` 참조.

## 4. 데이터 공백 — 우리 서비스에 없어서 새로 만들어야 하는 것

블로커를 메우려면 아래가 **선행**돼야 한다. "배선만 하면 됨"과 "모델부터 없음"을 엄격히 구분했다.

### D1. 배송·반품 정책 — **취소. 데이터 공백이 아니다**

이전 판단은 "`ChannelAccount`/`LegalEntity` 스코프 배송정책 모델 + 설정 UI +
`ChannelListing` 승격 마이그레이션" 이었다. **실측으로 전부 불필요해졌다** (§3 B1·B2).

- 값은 이미 WING 계정에 설정돼 있고 상품별로 변하지 않는다.
- 엑셀 양식에 배송·반품 컬럼이 없으므로 저장해도 실을 곳이 없다.
- `deliveryInfo` Json 파싱 스펙(이전에 "진짜 블로커" 로 지목)도 **더 이상 선결조건이 아니다.**
  출고소요시간은 `deliveryInfo` 를 파싱할 필요 없이 `출고리드타임`(col65) 상수로 끝난다.

남는 것은 **판매자 프리셋 상수 6개 + 확장의 일치 확인 단계** 뿐이다.
조직이 둘 이상 되면 그때 스코프를 올린다(§3 B1·B2 "미루는 것").

### D1-b. 등록상품명(판매자 관리용) 생성 규칙 — **신규**

라이브는 노출상품명과 별개로 `3000선인장딸깍키링` 형태를 쓴다. 접두 숫자 + 공백 제거로
**셀피아 상품명과 같은 형태**다. `WingProduct` 에 `productName` 하나뿐이라 표현 수단이 없고,
그 형태를 만드는 헬퍼도 저장소에 없다.

이건 단순 표기 문제가 아니라 **조인 축**이다. 셀피아 재고/발주 매칭이 이 형태의 이름에
의존하므로(`spec §8` 표기 차이 함정), 등록상품명을 셀피아 형태로 맞춰두면 D6 조인키 문제가
같이 얇아진다. 접두 숫자 파싱은 커버리지 61% 라 생성에만 쓰고 역파싱에는 쓰지 않는다.

### D2. 카테고리별 속성(attribute) 정의 소스 — **엔드포인트 미발견 (매핑만)**

`spec §3` 대로 속성 정의 API 를 못 찾았다. 다만 시드가 **두 개** 있다:

1. `extensions/coupang-ads-scraper/shared/coupang-catalog-collector.js:146-153` 이 쿠팡 API 의
   `item.attributes` 를 `{type, value}`(= `WingOption` 과 동일 형태)로 정규화해 수집한다.
   현재는 **경쟁/기존상품 읽기(import) 방향**이고 등록(write)과 배선되지 않았다.
2. **양식 `hidden` 시트**(`spec §9`) — `색상계열` 17종, 사이즈·패턴·소재·핏 열거.
   속성 **값** 후보는 이걸로 충분하다.

즉 미확보인 것은 "어떤 카테고리에 어떤 속성이 몇 개 붙는가" 라는 **매핑 하나**로 좁혀졌다.
→ 자사 기등록 상품의 `attributes` 를 카테고리별 프리셋 시드로 역이용하는 것이
API 재리버스보다 현실적이다(§6).

### D3. `displayCategoryCode ↔ 내부 categoryId` 매핑 — **규칙 미확보**

`77390 ↔ 6827` 매핑 규칙을 모른다(spec §2). 이것 없이는 `notice/form`·`certification`·
`delivery-charge-types` 를 카테고리별로 호출할 수 없다. **관측상 `id = code - 1000`(3/3)이나
단정 금지.** 라이브 재리버스 필요 → §7.

### D4. 브랜드·제조사 / 미성년자 / 부가세 / 병행수입 — **브랜드·제조사는 필드가 아니라 값이 틀렸다**

**먼저 값을 고쳐라. 모델링은 그 다음이다.** 라이브 기준은 브랜드 = 공란 + `브랜드없음` ☑,
제조사 = `해피프랜즈` 인데 프리셋은 `defaultBrand:'해피프랜즈'` / `defaultMaker:'kiditem'`
으로 **정확히 뒤바뀌어 있다**(`excel.ts:166-167`). 게다가 바로 위 주석이
"브랜드=해피프랜즈, 제조사=kiditem (실제 등록 상품 기준)" 이라고 **틀린 근거를 단언**하고 있어,
이걸 먼저 지우지 않으면 후속 수정이 계속 같은 방향으로 틀린다.
`excel.ts:184` 의 `maker || brand` 폴백은 양식 col7 설명("제조사를 알 수 없으면 브랜드를 입력")과
같은 취지라 유지해도 되지만, 브랜드를 공란으로 바꾸면 폴백도 같이 비므로 함께 정리한다.

미성년자·부가세·병행수입은 실측(전체 연령 / 과세 / 아님)과 코드 리터럴
(`excel.ts:199-201` `N`/`Y`/`N`)이 **일치한다.** 셋 다 양식상 `선택(기본값)` 이라 시급하지 않다.
다만 상품별 표현 수단이 없어 면세 품목이 섞이면 대응 불가인 것은 그대로다 —
카테고리 프리셋 필드로 승격(최소) 또는 `ProductPreparation.registrationInput` 필드 추가(정확).

### D5. sourcing → inventory 크로스도메인 포트 — **배선만 필요**

데이터·읽기 포트는 **이미 있다**(`SELLPIA_INVENTORY_SKU_READ_PORT`). 소비자는
`products/product-recipe-component-candidate.service.ts` 와
`channels/sellpia-recipe-evidence.adapter.ts` 둘뿐이고 **sourcing 참조 0건**이다.
`sourcing.architecture.spec.ts:92` 가 금지목록에 `inventory` 를 포함하므로
**직접 import 는 스펙 실패** — outbound cross-domain 포트가 유일한 적법 형태다.
선례: advertising 이 analytics 의 ABC 등급을 쓰려고 만든 `SELLPIA_ABC_GRADE_PORT`
(cross-domain port + `useExisting` 바인딩) 패턴을 그대로 복제한다.

### D6. 수집상품 ↔ 셀피아 SKU 조인키 — **미확정**

포트가 있어도 **무엇으로 조인할지가 안 정해졌다.** 수집상품(candidate)은 아직 매입 전이라
대응 SKU 가 없을 수 있다. `WingVariant` 리터럴에 `barcode`·`vendorItemCode` 키가 아예 없어
(`flow.ts:99-108`) 등록 후 재고 동기화도 이름매칭 휴리스틱에 영구 의존하게 된다.
필요: 바코드 우선 → 없으면 정규화 이름(`findByNormalizedNames`, SQL 구현 완료) 매칭 정책 확정.
메모리 `project_rocket_confirm_sellpia_stock_link` 의 키 함정(셀피아=legacyCode / 로켓=barcode)을 참고.

### D7. 이미지 규격 실측 — **검증기는 있고 배선만 없음**

`content_assets.width/height` 가 NULL 이라 크기로 못 거른다(spec §4).
그러나 **실측 검증기가 이미 존재한다**: `apps/server/src/ai/domain/thumbnail-image-spec.ts` 의
`parseImageDimensions`(sharp) + `deriveImageSpec` 이 `<1000px` fail, 1:1 비율 이탈 fail,
10MB 초과 fail 을 판정한다. 현재 **썸네일 AI 분석 도메인 전용**이고 WING 등록 경로에서
호출되지 않는다. → 신규 구현이 아니라 **재사용 배선**이다.
단 임계값이 저장소 안에서 엇갈린다(문서 "최소 500" vs 코드 "1000 미만 fail") — 통일 필요.

### D8. 원가·환율 기반 판매가 산정 — **역방향만 있음**

`rocket-pricing.ts`(`CNY_TO_KRW_RATE=190`, `unitCostFromCostCny`, `computeRocketPricing`),
`option-pricing-resolver.ts`(`costPrice ?? costCny*190`),
`sourcing-1688-new-product-model.ts:525`(`landedCostKrw`) 에 재료가 있으나
**전부 salePrice 를 입력으로 받는 소비자**다. 원가 → 판매가 산정기는 없다.
셀피아 `salePrice` 연결(D5)이 더 짧은 길이다.

### D9. 노출상품명 vs 등록상품명 — **"작은 갭" 판정 철회, D1-b 로 승격**

이전에는 "등록상품명은 WING 파생값에 맡긴다 / 노출 리스크 없음 / 우선순위 낮음" 으로 봤다.
실측 결과 **등록상품명은 파생값이 아니라 사람이 채워 넣은 별도 값**이었고
(`3000선인장딸깍키링`, 11/100), 그 형태가 셀피아 상품명과 같다. 조인 축이므로 D1-b 로 옮긴다.

양식 사실관계는 그대로다 — V4.6 `기본` 시트에 노출상품명 컬럼은 없고 `등록상품명`(col1)
한 칸뿐이며, 설명 원문이 "발주서에서만 사용" 이라고 못박는다(`spec §9`).
**즉 엑셀 경로가 그 칸에 노출용 긴 이름을 넣는 것은 양식 의도와 반대다.**

## 6. 설계 결정

### 하드코딩 프리셋을 어떻게 걷어낼 것인가

현재 물총 프리셋이 강제되는 이유는 세 겹이다. **하나만 고치면 안 된다.**

1. `WING_TOY_WATERGUN_PRESET` 이 저장소 유일한 `WingCategoryPreset` 이고,
   `flow.ts:69,131,177` 세 진입점이 전부 이를 기본 인자로 받으며,
   호출부 `page.tsx:178,205` 는 preset 을 넘기지 않는다.
2. 옵션명은 프리셋이 아니라 `flow.ts:100-103` 의 **문자열 리터럴**이 결정한다
   (`requiredPurchaseOptionTypes` 는 읽히지 않는 dead config).
3. 확장은 고시를 프리셋과 **무관하게** 처리한다(`fill.js:719` 고시 미구현).
   브랜드는 예외다 — `fill.js:617-620` 의 `selfMade` 고정이 **라이브 기준으로 정답**이므로
   프리셋을 소비하게 바꾸면 오히려 나빠진다. 프리셋 쪽 값을 라이브에 맞추는 것이 맞다(S1-b).

→ 걷어내는 순서: **(a) 프리셋 값을 라이브 기준으로 정정(S1-b) →
(b) 프리셋을 `categoryCode → preset` 맵으로 전환 →
(c) `flow.ts` 옵션 리터럴을 `preset.requiredPurchaseOptionTypes` 소비로 전환(dead config 활성화) →
(d) 확장의 고시·제조사 미구현 매핑 구현.**
(b)만 하면 아무것도 안 바뀌고, (a) 없이 (b) 를 하면 **틀린 값을 카테고리 수만큼 복제한다.**

### 카테고리별 옵션명·고시정보를 어떻게 동적으로 얻을 것인가

우선순위 순으로 세 갈래. 상위가 막히면 하위로 폴백한다.

| 방안 | 근거 | 리스크 |
|---|---|---|
| **A. 자사 카탈로그 역이용(권장 1순위)** | `coupang-catalog-collector.js:146-153` 이 이미 `item.attributes` 를 `{type,value}`(=`WingOption` 동형)로 수집한다. 자사 기등록 상품을 카테고리별로 묶어 **실제 쓰인 속성**을 프리셋 시드로 만든다 | 우리가 등록해 본 적 없는 카테고리는 커버 불가 |
| **B. `categoryId` 매핑 재리버스** | 매핑만 확보하면 `notice/form`·`certification`·`delivery-charge-types` 를 정식으로 호출 가능 | spec §2 에서 이미 실패. 라이브 재리버스 필요(§7 U4) |
| **C. 속성 정의 API 재탐색** | spec §3 의 미해결 항목 | `featureGroup` 이 A/B 플래그로 판명된 전례. 성공 보장 없음 |

**A 를 먼저 한다.** B/C 는 라이브 리버스 세션이 필요하고 결과가 불확실한 반면,
A 는 이미 저장소에 있는 코드의 방향만 바꾸는 일이다. A 로 커버 안 되는 카테고리는
**등록 차단**(카테고리 추론과 동일 정책)으로 처리하고 B/C 를 병행 조사한다.

### 이미지 규격 검증을 새로 만들지 않는다

`thumbnail-image-spec.ts` 의 `deriveImageSpec` 을 WING 등록 경로에 재사용한다(D7).
임계값이 문서(최소 500)와 코드(1000 미만 fail)로 엇갈리므로 **하나로 통일**하고,
`content_assets.width/height` NULL 문제는 업로드 시 dimension 기록 또는
확장에서 `createImageBitmap` 실측 둘 중 하나로 정한다.

### 두 경로를 유지할 것인가

현재 "일괄"이 오히려 자동화가 덜 됐다(다운로드 폴더까지). 두 경로를 영구 유지하면
S2~S6 의 모든 변경을 두 번 구현해야 한다. **`candidateToWingProduct` 가 두 경로의 유일한
공통 변환기**이므로 값 매핑은 여기 모으고, 경로별 차이는 "쓰기 어댑터"(엑셀 셀 / 폼 DOM)로만
남기는 방향을 권한다. 경로 통합·폐기 여부는 별도 판단이 필요하다.

### 검증 없이 단정하면 안 되는 것

- **"medium 신뢰도 카테고리 자동적용이 안전한가"** — 실측 38%다. 현재는 "제출은 사람이"
  라는 전제로 정당화되지만(spec §8), S8 에서 제출을 자동화하면 이 전제가 무너진다.
  **제출 자동화와 medium 자동적용은 동시에 존재할 수 없다.**
- **"상세페이지 렌더 폭이 780px 이다"** — 요청 파라미터일 뿐 출력 보장이 아니다.
  `detail-page-rasterization.service.ts` 의 `getContentClip` 은 콘텐츠 바운딩박스 기준이라
  중앙정렬 컨테이너가 있으면 780 미만이 나오고, 응답의 `outputWidth: 780` 은
  **측정값이 아니라 입력값 에코**다. 어느 층에도 결과 폭 검증이 없다.
- **"라이브와 값이 같으니 코드가 맞다"** — 이번 실측에서 **우연히 맞은 항목이 여럿**이다.
  옵션명 `색상`/`수량`(물총 하드코딩인데 열쇠고리도 마침 동일), 수량 `1`, 재고 `999`,
  성인상품 `N` / 과세 `Y` / 병행수입 `N`. 전부 **표본 1건 일치**이고 근거는 없다.
  카테고리가 동적으로 해석되는 이상 다음 상품에서 깨진다. **"아직 반례가 없다" 와
  "맞다" 를 구분해서 기록하라.**
